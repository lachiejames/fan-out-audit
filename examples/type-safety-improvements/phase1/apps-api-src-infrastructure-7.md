# Audit: apps-api-src-infrastructure-7

**Files inspected**: 8
**Findings**: 4

## Summary

This batch covers queue service files: attachment-embedding, background-jobs adapter, content-hydration, domain-events, knowledge-source-import-batch, knowledge-source-import, knowledge-source-quick-import, and knowledge-source-searchable-archive queues. Most queue service files are one-to-three lines generated by the factory and have no type-safety issues. The findings are concentrated in `background-jobs.adapter.ts`.

## Findings

### Finding 1: `enqueueWebhookJob` accepts `data: unknown`, bypassing job type safety

- **File**: `apps/api/src/infrastructure/queues/background-jobs.adapter.ts:153`
- **Category**: missing-strict-typing
- **Impact**: high
- **Description**: `enqueueWebhookJob` is typed with `data: unknown`. All other enqueue methods in `BackgroundJobsAdapter` use strongly-typed job data types (e.g. `WorkItemGenerationJob`, `PredictionGenerationJob`). The `unknown` type means the compiler cannot verify that callers pass the correct webhook job payload shape. At runtime the data is passed directly to BullMQ which serialises it to JSON — a structurally wrong payload would be silently enqueued and only fail when the processor tries to destructure it.
- **Suggestion**: Import `WebhooksJob` (or equivalent) from `@/shared/constants/queues/webhooks.constants` and type `data: WebhooksJob` in the parameter. If webhook jobs are heterogeneous, define a discriminated union `type WebhookJobPayload = SlackWebhookJob | GmailWebhookJob | ...` and use that.
- **Evidence**:

```typescript
async enqueueWebhookJob(params: { name: string; data: unknown; jobId?: string }): Promise<void> {
  const { data, jobId, name } = params;
  await this.webhooksQueue.queue.add(name, data, jobId ? { jobId } : undefined);
}
```

### Finding 2: `WebhooksQueueService` uses untyped `createQueueService()` (no generic)

- **File**: `apps/api/src/infrastructure/queues/webhooks-queue.service.ts:3`
- **Category**: missing-strict-typing
- **Impact**: medium
- **Description**: `createQueueService({ queueName: WEBHOOKS_QUEUE })` is called without a type parameter. The factory defaults to `T = unknown`, giving `WebhooksQueueService.queue` type `Queue<unknown>`. This means `queue.add(name, data)` accepts any value for `data` with no compile-time verification, consistent with Finding 1 above.
- **Suggestion**: Define a webhook job union type in `webhooks.constants` and pass it as the generic: `createQueueService<WebhookJob>({ queueName: WEBHOOKS_QUEUE })`.
- **Evidence**:

```typescript
export const WebhooksQueueService = createQueueService({ queueName: WEBHOOKS_QUEUE });
```

### Finding 3: `PatternExtractionQueueService` also uses untyped `createQueueService()`

- **File**: `apps/api/src/infrastructure/queues/pattern-extraction-queue.service.ts:5`
- **Category**: missing-strict-typing
- **Impact**: medium
- **Description**: Same issue as Finding 2. `createQueueService({ queueName: PATTERN_EXTRACTION_QUEUE })` is called without a type parameter even though `PatternExtractionJob` is defined in `pattern-extraction.constants`. This gives the queue's `add()` method an `unknown` data parameter.
- **Suggestion**: Import `PatternExtractionJob` from `@/shared/constants/queues/pattern-extraction.constants` and pass it as the generic.
- **Evidence**:

```typescript
export const PatternExtractionQueueService = createQueueService({ queueName: PATTERN_EXTRACTION_QUEUE });
```

### Finding 4: `createQueueService` factory uses `any[]` in its return type signature

- **File**: `apps/api/src/infrastructure/queues/queue-service.factory.ts:22`
- **Category**: any-usage
- **Impact**: medium
- **Description**: The `createQueueService` factory returns `new (...args: any[]) => QueueServiceInstance<T>`. The `any[]` constructor args type is necessary because the inner `QueueService` class uses `@InjectQueue` which NestJS resolves at runtime, but it means the returned class constructor accepts any arguments without type checking. This is a standard NestJS factory pattern limitation but it is worth noting.
- **Suggestion**: This is largely unavoidable with NestJS DI, but the `any[]` could be narrowed to `[Queue<T>]` to at least document the expected single argument, even if NestJS ignores the signature at injection time.
- **Evidence**:

```typescript
export function createQueueService<T = unknown>({
  queueName,
}: {
  queueName: string;
}): new (
  ...args: any[]
) => QueueServiceInstance<T> {
```
